江湖钓客三端适配与三语国际化实施计划

# 一、背景与目标

江湖钓客此前以手机页面为主要设计基准，页面在桌面宽屏下会被直接放大。首页功能入口因此出现细长竖条、留白失衡、文字密度过高等问题；iPad 与 Web 也缺少独立的导航和内容宽度策略。

本次改造按“布局优先、国际化随后、全应用分批完成”的顺序执行，目标是：

  1. 统一 iPhone、iPad、Web 的响应式规则。
  2. 优先修复首页入口比例和全局导航。
  3. 建立简体中文、English、한국어 三语基础设施。
  4. 让语言切换即时生效并持久化。
  5. 用自动化测试和中文硬编码基线约束后续开发。

# 二、改造前的问题

桌面端首页的功能入口没有撑满网格单元，卡片比例和内容区留白明显失衡。

主要问题包括：

  • 分散使用多个断点，页面之间缺少统一规则。
  • 宽屏继续按手机设计尺寸缩放，组件比例不稳定。
  • 首页 InkPressable + Stack 未撑满网格单元。
  • 手机、平板、桌面共用同一种导航结构。
  • 文案直接写在组件中，缺少统一翻译和语言状态。
  • 英文、韩文较长时容易挤压图标或改变组件高度。

# 三、响应式设计方案

# 3.1 四级断点

模式 宽度 主要终端 导航与布局
compact < 600 iPhone 单列内容、底部导航、首页 2 列入口
medium 600–1023 iPad 竖屏 紧凑 Navigation Rail、首页 3 列入口
expanded 1024–1439 iPad 横屏、普通 Web 完整侧栏、受控内容宽度、首页 4 列入口
wide >= 1440 桌面 Web 完整侧栏、最大宽度容器、首页最多 6 列入口

布局由容器宽度决定，字号、圆角和按钮高度保持稳定。内容区最大宽度分别限制为 560 / 920 / 1240 / 1480，避免页面在超宽屏无限拉伸。

# 3.2 三端页面结构

iPhone

  • 保留底部导航。
  • 首页功能入口使用 2 列。
  • 弹窗优先使用 Bottom Sheet。
  • 控件保持足够触控面积，避免长文本挤压图标。

iPad 竖屏

  • 使用紧凑 Navigation Rail。
  • 首页入口使用 3 列。
  • 内容区居中并限制最大宽度。

iPad 横屏

  • 切换为完整侧栏。
  • 主内容获得更稳定的阅读宽度。
  • 后续设备详情可在右侧增加遥测或控制面板。

Web

  • 使用左侧导航。
  • 页面居中并限制最大宽度。
  • 首页入口随可用空间切换 4 至 6 列，不再出现细长竖条。

# 四、三语国际化方案

# 4.1 语言切换流程

应用启用 Flutter gen_l10n，维护以下资源：

  • app_zh.arb
  • app_en.arb
  • app_ko.arb

首次启动读取系统语言，不支持的语言回退到简体中文。设置页提供“跟随系统、简体中文、English、한국어”，用户选择保存在 SharedPreferences，切换后立即刷新，不要求重新登录。

韩文 iPad 实际效果：

韩文使用随应用打包的 Noto Sans KR，并为 Flutter Web 提供离线字体回退资源，避免断网或字体 CDN 不可用时出现方框字。

# 4.2 客户端与服务端数据流

客户端根据当前有效语言发送请求头：

Accept-Language: zh-CN
Accept-Language: en-US
Accept-Language: ko-KR

FastAPI 中间件统一解析语言，并返回实际采用的语言：

Content-Language: zh-CN

用户昵称、设备自定义名称、社区正文等用户数据保持原文。日期、数字、温度、长度和重量将在后续业务页面迁移时统一接入区域格式化器，首轮仍使用公制单位。

# 五、实施内容

本轮已完成：

  • 新增统一 AppBreakpoint 与 ResponsiveLayout。
  • 首页入口改为手机 2 列、iPad 3 列、Web 4 至 6 列。
  • 修复首页入口未撑满网格导致的比例畸形。
  • iPhone 使用底部导航，iPad 使用 Navigation Rail，Web 使用完整侧栏。
  • 全局页面增加最大宽度约束。
  • 新增中、英、韩 ARB 资源和 LocaleController。
  • 设置页新增语言选择，并持久化用户设置。
  • 导航、首页、设置、设备中心等核心界面完成三语迁移。
  • 请求自动携带 Accept-Language，服务端返回 Content-Language。
  • 增加 Flutter 布局测试、语言测试和后端语言头测试。
  • 增加中文界面硬编码基线检查，阻止后续新增未登记中文。

# 六、分阶段迁移计划

# 第一阶段：全局与核心页面

状态：本轮已完成。

  • 登录与全局壳层
  • 首页
  • 设备中心与设备详情公共组件
  • 设置与语言切换
  • 发现、商城、个人中心一级入口

# 第二阶段：核心任务流

  • 钓场列表与详情
  • 智能钓箱、智能钓伞、智能钓台的控制与遥测
  • 商城商品、购物车和结算
  • 通用错误、空状态、加载状态
  • 日期、数字和设备单位格式化

# 第三阶段：剩余业务页面

  • 社区与鱼获发布
  • 个人中心二级页面
  • 演示数据与首页推荐内容
  • 后端可运营内容的多语言模型

当前扫描基线为 3364 处中文界面候选。该数字包含业务内容、演示数据和部分非界面文本，后续按页面逐批确认和迁移；自动化脚本会阻止数量继续增加。

终端 基准尺寸 首页 导航 中英韩 状态

完整交付标准：

  • 无溢出、遮挡、异常换行、横向滚动和导航覆盖。
  • 英文、韩文长文本不改变固定控件高度。
  • 横竖屏切换保持当前路由、设备选择和表单状态。
  • 每个断点完成登录、首页、设备控制、钓场、商城和设置流程。
  • Flutter 分析、测试、后端测试和 Web Release 构建全部通过。

# 八、测试与验证

本轮代码验证结果：

  • 后端测试：44 passed
  • Flutter 测试：20 passed
  • flutter analyze：无问题
  • Web Release：构建成功
  • 中文硬编码基线：3364 <= 3364
  • Web、iPad 横屏、iPad 竖屏、iPhone 已进行实际浏览器截图检查

# 九、风险与后续优化

  1. 全应用中文存量较大，必须按业务页面迁移，避免一次性替换造成语义错误。
  2. 服务端运营内容需要补充多语言字段、回退规则和后台编辑能力。
  3. 设备遥测中的单位与告警文案要与固件协议一起定义，不能只在前端翻译。
  4. 需要继续补充三端截图测试和横竖屏状态保持测试。
  5. 后续新增页面必须使用统一断点和本地化资源，不再自定义临时宽度规则。

本轮先解决最影响观感和可用性的首页比例、全局导航与三语基础设施。接下来将按设备控制、钓场、商城、社区的顺序继续迁移，并在每一批完成后更新验收矩阵和实施记录。
