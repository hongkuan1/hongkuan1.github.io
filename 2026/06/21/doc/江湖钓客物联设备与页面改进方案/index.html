江湖钓客物联设备与页面改进方案

发表于2026-06-21本文字数7.9k字阅读时长7 分钟
  • Yume Shoka
首页智能垂钓

本文记录江湖钓客 App 在 2026 年 6 月的产品、页面、代码和物联设备专项审计结果，并给出可执行的改造路线。

本次审计实际启动了 Flutter Web 与 FastAPI 后端，检查了登录、首页、钓场、补给、我的、鱼圈、设置、购物车、结算和鱼获发布等页面，同时核对了路由、脚本、数据库模型、设备接口和大文件结构。

结论先说：当前系统已经有较完整的垂钓服务原型，也已经具备设备、遥测、告警和固件的数据骨架，但智能钓箱、智能钓伞、智能钓台还没有形成独立、可控制、可扩展的产品闭环。

# 一、当前产品状态

# 1. 页面主线已经形成

当前主流程包括：

  1. 首页判断今天是否适合出钓。
  2. 钓场页选择钓点、查看路线和预约能力。
  3. 补给页购买装备、耗材和服务。
  4. 开钓入口记录鱼获和现场信息。
  5. 我的页面查看用户资产和设备摘要。

真实后端启动后，首页可以读取南京试点城市数据，设备接口能够返回 5 台设备，其中 4 台在线、1 台低电量。

# 2. 设备目前停留在摘要层

“我的”页面已经显示设备数量、在线状态、低电量和异常提醒，但点击后没有完整的设备中心、单设备详情、远程控制和维护页面。

已有后端能力：

能力 当前状态
设备列表 已有 /api/v1/devices
单设备详情 已有 /api/v1/devices/{device_id}
遥测快照 已有 /telemetry
设备告警 已有 /alerts
固件信息 已有 /firmware
设备绑定与解绑 缺失
设备能力描述 缺失
控制命令与回执 缺失
场景联动 缺失
实时事件推送 缺失

因此这不是从零开始，而是需要把“只读设备数据”升级为“可操作的设备产品”。

# 二、目前需要解决的问题

# 1. 桌面 Web 没有真正的桌面布局

当前 Web 在宽屏中固定为约 390px 的手机画布，居中放在深色背景上。它适合演示手机界面，但没有利用桌面空间，也无法承担设备多面板、遥测曲线和批量控制。

调整方向：

  • 手机端保持单列任务流。
  • 平板端采用设备列表与详情的双栏结构。
  • 桌面端采用左侧导航、中间设备列表、右侧实时详情的三栏结构。
  • 不再把桌面端简单理解为“手机壳放大展示”。

# 2. 快捷入口比例失衡

首页、钓场和补给页的快捷入口被拉成细长竖条，文字和图标集中在下方，上部留白较多。

调整方向：

  • 手机端使用 2 列紧凑功能块。
  • 桌面端使用横向工具栏或 3 至 4 列功能网格。
  • 每个入口只保留一个动作名和一条状态，不重复解释。
  • 设备入口需要显示异常数量或最近同步状态，而不是只写“同步”。

# 3. 设备入口能看到，但不能完成任务

用户现在只能看到“4 台在线、1 台低电量”，无法继续完成：

  • 查看哪台设备异常。
  • 查看历史数据。
  • 调节设备。
  • 确认控制是否执行成功。
  • 更新固件。
  • 绑定新设备。
  • 创建开钓、夜钓和收竿联动场景。

设备页面必须从“展示状态”升级为“完成设备任务”。

# 4. 页面文字密度仍然偏高

目前多个页面同时出现标题、副标题、状态、解释和按钮，重要信息与辅助说明视觉权重接近。

调整原则：

  • 一个区块只回答一个问题。
  • 首屏优先显示状态、风险和主要动作。
  • 解释内容放入二级页面、帮助弹层或信息提示。
  • 状态使用短标签，不用完整句子。
  • 危险控制使用确认流程，不靠大段警告文字制造安全感。

# 三、设备中心目标结构

设备中心采用“设备驾驶舱 + 场景联动”的组合：

  1. 默认首页展示整体健康度、异常设备和三个重点设备。
  2. 场景页以开钓、夜钓、收竿组织多个设备动作。
  3. 每个设备进入独立详情页。
  4. 告警中心统一处理异常、低电量、离线和校准问题。
  5. 新增设备入口支持扫码、蓝牙发现和手动选择型号。

建议新增路由：

路由 用途
/devices 设备中心
/devices/:deviceId 单设备详情
/devices/add 新增与绑定设备
/device-scenes 场景联动
/device-alerts 告警中心
/device-commands/:commandId 控制命令结果

# 四、三类重点设备页面设计

三张页面保持统一骨架：

  • 顶部：设备名称、在线状态、电量、信号。
  • 一级标签：状态、控制、自动化、维护。
  • 中部：设备专属核心控制。
  • 下部：趋势、事件、告警、固件和诊断。
  • 危险操作：长按确认或二次确认。

# 1. 智能钓箱

核心任务是保鲜、电源和收纳安全。

主要功能：

  • 当前温度与目标温度。
  • 制冷模式：低、中、高。
  • 箱锁控制，长按锁箱。
  • 箱内照明。
  • USB 电源输出。
  • 鱼获存放重量和保鲜时长。
  • 24 小时温度趋势。
  • 温度偏高、箱盖未关、电量不足告警。
  • 固件升级和传感器校准。

# 2. 智能钓伞

核心任务是防晒、防雨和防风安全。

主要功能：

  • 风力、阵风、紫外线和降雨概率。
  • 开伞、收伞。
  • 左右倾角调节。
  • 防风阈值。
  • 超过阈值自动收伞。
  • 太阳跟随模式。
  • 降雨响应。
  • 低电量和电池健康。
  • 最近自动化事件。

收伞会改变设备机械状态，应使用长按确认，并明确显示“下发中、已收伞、执行失败”。

# 3. 智能钓台

核心任务是调平、承重和岸边安全。

主要功能：

  • 当前倾斜角度。
  • 一键调平。
  • 前左、前右、后左、后右四腿微调。
  • 负载分布。
  • 稳定性判断。
  • 安全锁。
  • 紧急停止。
  • 校准历史。
  • 固件与执行器诊断。

紧急停止需要与普通操作分区，防止误触；但不能用过度夸张的警示占据整个页面。

# 五、系统建模

现有 SmartDevice 统一表可以保留，但不应继续把不同设备的专属能力都塞入 extra_data。

建议把设备域拆成四个问题：

  1. 设备是什么：产品型号、协议和固件。
  2. 设备能做什么：属性、命令、范围和危险等级。
  3. 设备发生了什么：遥测、事件和告警。
  4. 用户下达了什么：命令、回执、超时和审计。

建议核心实体：

实体 责任
DeviceProductModel 型号、厂商、设备类型、协议版本
DeviceInstance 用户实际绑定的设备
DeviceCapability 属性、命令、参数约束和危险级别
DevicePropertySnapshot 当前状态快照
TelemetryPoint 时序遥测数据
DeviceCommand 一次用户控制请求
CommandReceipt 设备执行结果
DeviceAlert 异常及处理状态
AutomationScene 多设备联动场景
SceneAction 场景内的设备动作
FirmwareRelease 固件版本与升级策略

# 新设备如何扩展

新增智能夜钓灯、增氧设备、探鱼器或新型号时：

  1. 注册新的 DeviceProductModel。
  2. 配置能力描述。
  3. 复用通用状态、告警、固件和命令页面。
  4. 仅新增必要的专属控制模块。

不应为每个设备复制一套数据库表、接口和状态管理代码。

# 六、设备控制工作流

远程设备控制不能把“用户点击按钮”直接当成“设备已经执行”。

动态过程如下：

标准状态建议：

draft → awaiting_confirmation → queued → sent → acknowledged → succeeded

异常分支：

rejected / failed / timed_out / cancelled / rolled_back

建议新增接口：

方法 接口 用途
GET /devices/{id}/capabilities 获取设备能力
POST /devices/bind 绑定设备
DELETE /devices/{id}/binding 解绑设备
PATCH /devices/{id}/settings 修改设备设置
POST /devices/{id}/commands 下发控制命令
GET /device-commands/{id} 查询命令结果
GET /devices/events SSE 或 WebSocket 实时事件
POST /device-scenes 创建联动场景
POST /device-scenes/{id}/execute 执行场景
POST /devices/{id}/firmware-upgrades 发起 OTA

# 2. 中文注释怎么加

需要添加中文注释的地方：

  • 领域对象的业务含义和边界。
  • 危险设备命令的安全约束。
  • Mock、真实 API 与硬件回执的切换逻辑。
  • 不直观的状态机和失败回滚。
  • 平台差异、兼容逻辑和历史原因。

不需要注释的地方：

  • 简单赋值。
  • 普通 Widget 布局。
  • 函数名已经能说明用途的代码。
  • 重复描述代码表面行为的注释。

目标是帮助人工和 AI 快速理解“为什么这样设计”，而不是把每行代码翻译成中文。

# 八、无关文件与废案处理

# 可以直接清理

  • assets/fonts 下 4 个未在 pubspec.yaml 注册的 TTF，约 14MB。
  • .dart_tool、build、虚拟环境、缓存和本地数据库等生成物。
  • 项目根目录误生成的空 SQLite 文件。
  • 已被当前水墨组件替代且没有引用的旧视觉组件。

# 需要确认后删除

静态可达性检查发现以下代码没有从 main.dart 主路径引用：

  • core/platform/avatar_picker.dart
  • core/utils/discard_dialog.dart
  • core/utils/number_formatter.dart
  • features/explore/data/explore_models.dart
  • features/explore/data/explore_repository.dart
  • features/mall/data/mall_repository.dart
  • features/profile/data/ops_models.dart
  • features/profile/data/ops_repository.dart
  • shared/widgets/app_list_section.dart
  • shared/widgets/app_visuals.dart

删除前仍需逐个确认是否属于即将恢复的功能。

home_location_reader_native.dart 和 home_location_reader_web.dart 是条件导入，不属于废案，必须保留。

# 命名需要统一

脚本和后端仍混用“智友”“渔趣”“江湖钓客”。建议：

  • 用户可见名称统一为“江湖钓客”。
  • 包名、数据库名和兼容字段可以暂时保持。
  • 修改内部名称时提供迁移说明，避免部署脚本和环境变量失效。

# 九、页面响应式修改方案

# 手机

  • 单列页面。
  • 首屏只放状态、风险和主操作。
  • 次要数据通过标签页和折叠区进入。
  • 底部控制区固定但不遮挡内容。

# 平板

  • 左侧设备列表，右侧设备详情。
  • 控制和趋势可以同时显示。
  • 场景联动使用双栏编排。

# 桌面 Web

  • 最大内容宽度建议 1180–1280px。
  • 左侧为设备与场景导航。
  • 中间为设备列表或控制面板。
  • 右侧为实时遥测、告警和命令日志。
  • 不再强制使用 390px 手机容器。

# 十、研发优先级

# P0：清理与护栏

  • 删除确定无引用的资源和废案。
  • 拆分超大页面。
  • 统一命名。
  • 补充路由、设备契约和启动脚本测试。

# P1：设备中心基础版

  • 设备中心、设备详情和新增设备路由。
  • 三类重点设备详情页。
  • 接入现有遥测、告警和固件接口。
  • 建立设备能力模型和扩展位。

# P2：控制闭环

  • 命令、回执和状态机。
  • 模拟硬件回执。
  • 危险操作确认。
  • 失败、超时、重试和审计。

# P3：场景与真实硬件

  • 开钓、夜钓、收竿场景。
  • MQTT、BLE 网关或厂商协议。
  • OTA、诊断和灰度升级。
  • 实时指标与告警监控。

# 十一、验收标准

完成 P0 和 P1 后，至少满足：

  1. 手机、平板和桌面没有明显比例畸形。
  2. 用户能从首页或“我的”进入设备中心。
  3. 用户能查看全部设备和异常设备。
  4. 智能钓箱、钓伞、钓台都有独立详情页。
  5. 页面使用真实设备接口，失败时明确标注 Mock 回退。
  6. 新增设备类型无需复制设备基础架构。
  7. 单个页面文件尽量控制在 300–600 行。
  8. 危险控制有确认、回执、超时和失败状态。
  9. 设备控制过程可以追踪到 command_id。
  10. Flutter 静态检查、测试和后端接口测试全部通过。

# 最终建议

优先完成 P0 清理与护栏 + P1 设备中心基础版。

这两个阶段能把当前“功能丰富但设备主线不突出”的原型，调整为“设备价值明确、页面可操作、后续硬件可扩展”的产品骨架。之后再进入控制闭环和真实硬件协议，返工成本会明显更低。
