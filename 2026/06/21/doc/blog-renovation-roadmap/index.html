博客现状体检与改造计划

发表于2026-06-21本文字数4k字阅读时长4 分钟
  • Yume Shoka
首页文档

这不是一次单纯的换皮，而是把博客改造成一个 可阅读、可互动、可观测、可持续维护 的个人知识站。

# 一句话结论

博客当前已经完成从 Shoka 到 ShokaX 的主体迁移，但仍处于“新主题已经上线，旧资产和旧配置尚未收口”的过渡状态。

最值得优先解决的不是继续堆动画，而是先恢复统计与音乐等基础反馈，清理半启用功能和重复资产，再升级互动看板娘与首页信息架构。

# 当前资产盘点

项目 当前状态 判断
内容 审计时 26 篇，本文发布后共 27 篇 内容基础已形成
当前主题 ShokaX 0.5.4，约 252MB 正在使用，包含独立依赖目录
旧主题 Shoka 0.2.5，约 2.7MB 已不使用，仍被 Git 跟踪
生成目录 public/ 约 34MB 存在名称带 2 的重复生成物
部署目录 .deploy_git/ 约 109MB 同样存在重复文件
根依赖 node_modules/ 约 476MB 本地开发依赖
站点统计 插件已安装，但配置关闭 功能可恢复
音乐播放器 模块开启，歌单为空 按钮存在但无法初始化
看板娘 Koharu + 旧 Live2D Widget 可运行，但年代较久、互动有限
评论系统 Waline、Twikoo 均关闭 “最新评论”区域缺少真实数据来源

# 已确认的问题

# P0：安全与发布可靠性

  1. 根配置中存在 Algolia adminApiKey 明文，不适合继续保存在公开仓库。
  2. _config.yml 的 sitemap.path 使用 Tab 缩进，不符合标准 YAML。
  3. public/、.deploy_git/ 和根目录存在大量重复文件，例如 index 2.html、db 2.json、db 3.json。
  4. 当前 Git 工作区包含大量生成物变化，源码改动与生成结果混在一起，容易误提交或误删除。

目标： 密钥退出仓库、配置可校验、构建结果可重复、部署目录可随时重建。

# P1：用户可感知的功能缺口

# 统计文字没有显示

hexo-symbols-count-time 已安装，ShokaX 也包含对应模板，但项目配置为：

footer:
  count: false

post:
  count: false

这不是主题不支持，而是站点覆盖配置主动关闭了站点总字数、阅读时长和文章字数。

# 音乐按钮无法使用

音乐模块已经开启：

modules:
  player: true

但全局 audio: 为空。播放器初始化代码检测不到歌单后会直接返回，因此页面保留按钮，却没有可播放内容。

此外，当前播放器只读取每个列表的第一个地址，并依赖第三方音乐解析接口。后续需要同时处理歌单配置、接口可用性、失败提示和移动端交互。

# 看板娘互动能力较弱

当前方案是：

  • hexo-helper-live2d
  • live2d-widget-model-koharu
  • 本地 L2Dwidget 脚本

它能显示角色和固定文本，但缺少现代交互层，例如点击菜单、随机对话、文章推荐、返回顶部、换装/换模型、状态记忆和移动端降级。

# P2：体验与内容结构

  1. 首页依赖随机动漫图片 API，视觉不稳定，也会带来加载速度和可用性风险。
  2. “随机文章”和“最新评论”同时开启，但评论后端关闭，形成空模块。
  3. 搜索依赖 Algolia，却没有明确的索引更新流程；Pagefind 已配置但关闭。
  4. 多篇旧文章存在分类字段拼写错误、Front Matter 风格不一致和内容时效问题。
  5. 首页目前缺少“站点状态、最近更新、重点项目、改造进度”等能说明博客价值的信息。

# P3：工程冗余

  1. themes/shoka 与 themes/shokax 双主题共存。
  2. ShokaX 自带 .git、node_modules 和独立锁文件，使主题目录膨胀到约 252MB。
  3. 根项目仍保留旧主题时代的依赖，例如 Landscape、Heroku deployer 和部分不再使用的渲染/搜索组件。
  4. temp_*.js、temp_*.html、多个 db*.json 没有统一清理规则。
  5. public/ 被 Git 跟踪，导致每次构建产生大量无关差异。

# 改造后的目标体验

用户来到博客后 期望体验
首次进入首页 立即知道“我是谁、博客写什么、最近在做什么”
阅读文章 能看到字数、预计阅读时间、更新时间和目录进度
点击音乐按钮 有明确歌单、播放状态、失败提示和音量控制
与看板娘互动 能点击唤出菜单、随机对话、推荐文章，并可关闭
浏览站点 图片和字体加载稳定，移动端不遮挡正文
查看首页 能看到改造进度与最新成果，而不是只有文章流
维护博客 只提交源码，构建和部署结果能够一键重建

# 分阶段改造路线

# 第一阶段：恢复基础反馈

验收： 文章页有统计文字；音乐按钮至少可以稳定播放一首歌；控制台无播放器初始化错误。

# 第二阶段：升级互动看板娘

验收： 桌面端可互动，移动端不遮挡内容，关闭状态能够被记住。

# 第三阶段：收口主题与依赖

验收： 全新克隆后只需一次安装和一次构建即可得到相同站点。

# 第四阶段：首页产品化

验收： 首页首屏能传达博客定位，核心入口在桌面和手机上都易于发现。

# 第五阶段：质量与持续维护

验收： 构建失败会被及时发现，外部依赖失效不会让核心阅读功能不可用。

# 推荐实施顺序

统计文字 → 最小歌单 → 播放器失败处理 → 看板娘替换 → 旧主题清理 → 首页改版。

这个顺序优先修复读者能直接感知的问题，同时把风险较高的主题和部署清理放在功能稳定之后。

# 不做什么

  • 不为了“炫酷”一次性叠加大量动画。
  • 不让看板娘、音乐或烟花遮挡阅读内容。
  • 不同时维护两套搜索、两套评论或两套主题。
  • 不继续把密钥、缓存、构建产物和源码混在同一个提交里。
  • 不在没有回滚方案的情况下直接删除旧主题。

# 进度看板

阶段 状态 说明
现状审计 已完成 已定位统计、音乐、看板娘与工程冗余问题
基础反馈 待开始 统计与音乐是第一批改造
看板娘升级 待评估 先确认模型授权与移动端策略
主题收口 待开始 删除前先做依赖核对
首页产品化 待开始 在基础功能稳定后进行
持续质量 长期进行 构建、链接、外部服务定期检查
