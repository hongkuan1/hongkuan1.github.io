江湖钓客物联设备中心改造执行完成报告

这篇文章是《江湖钓客物联设备与页面改进方案》的执行结果记录。

本轮改造已经完成批准范围内的产品、页面、接口、数据模型和多端适配：智能钓箱、智能钓伞、智能钓台不再只是“我的设备”中的一行摘要，而是形成了可以查看状态、远程控制、执行场景、追踪命令和处理维护任务的完整设备中心。

代码已经提交到 hk2012/zhiyou 仓库的 codex/iot-device-center 分支。

# 一、改造结果

本轮完成了五个核心目标：

  1. Web 不再固定显示为 390px 手机画布，桌面、平板和手机均使用真实响应式布局。
  2. 建立统一设备中心，集中处理设备健康、筛选、异常、场景和新增设备。
  3. 为智能钓箱、智能钓伞、智能钓台分别建立独立详情和控制能力。
  4. 建立设备命令、二次确认、状态流转、执行回执和时间线。
  5. 建立可扩展设备模型，为后续 MQTT、BLE 和厂商协议接入保留清晰边界。

# 改造前

改造前的桌面 Web 仍然是手机尺寸居中展示，宽屏空间没有被利用。

设备入口只能显示在线数量、低电量和告警摘要，用户无法继续进入单设备任务。

# 改造后

桌面端现在使用左侧应用导航、中间设备列表、右侧实时预览的三栏结构。常用动作保持在首屏，详细内容按任务逐层展开。

平板端采用双栏结构，同时保留设备列表和当前设备信息。

手机端回到单列任务流，只展示当前决策需要的信息，避免把桌面面板硬塞进小屏幕。

# 二、页面与交互调整

# 1. 信息密度重新分层

页面不再同时堆叠标题、副标题、状态说明和多个按钮，而是按以下层级展示：

  • 第一层：设备是否在线、是否安全、是否需要处理。
  • 第二层：当前最常用的控制动作。
  • 第三层：趋势、自动化和维护信息。
  • 第四层：命令编号、执行时间和故障详情。

用户进入页面后先得到结论，再决定是否继续查看数据。

# 2. 操作入口统一

设备中心统一提供：

  • 在线、离线、低电量和告警筛选。
  • 开钓、夜钓、收竿三个默认场景。
  • 蓝牙搜索和手动添加设备流程。
  • 设备异常入口和最近命令入口。
  • 手机、平板、桌面一致的任务语义。

# 3. 高风险动作增加确认

智能钓伞收伞、智能钓台急停等操作不能只依赖一次点击。系统会先说明影响，再由用户确认下发。

命令提交后，界面不直接假设设备已经成功执行，而是展示命令编号和状态时间线。

# 三、三个核心设备详情

三个设备共用“状态、控制、自动化、维护”四个任务标签，但控制内容根据真实使用场景分别设计。

# 1. 智能钓箱

智能钓箱详情已经支持：

  • 箱内温度查看和目标温度调节。
  • 制冷模式切换。
  • 箱锁、照明和 USB 电源控制。
  • 鲜度状态、温度趋势和异常提示。
  • 固件、校准和维护入口。

# 2. 智能钓伞

智能钓伞详情已经支持：

  • 风速、降雨、紫外线和伞体状态展示。
  • 开伞、收伞及危险动作二次确认。
  • 倾角调节、强风阈值和自动收伞。
  • 追日、遇雨响应和事件记录。

# 3. 智能钓台

智能钓台详情已经支持：

  • 水平度、载荷和稳定性展示。
  • 自动调平。
  • 四条支腿独立微调。
  • 安全锁和急停。
  • 异常状态与维护入口。

# 四、系统建模与扩展方式

本轮没有把三个设备写成三个互不相关的孤立页面，而是建立了统一设备能力模型。

新增设备时主要完成三件事：

  1. 声明设备能力，例如温度、照明、锁、角度或支腿。
  2. 为能力配置合适的控制组件和安全规则。
  3. 实现对应的协议适配器。

这样新增智能鱼护、智能增氧器或水质传感器时，不需要重新复制一套设备中心。

# 后端新增能力

本轮新增或完善了以下接口：

接口 作用
GET /api/v1/devices/{id}/capabilities 获取设备能力
POST /api/v1/devices/bind 绑定设备
DELETE /api/v1/devices/{id}/binding 解绑设备
PATCH /api/v1/devices/{id}/settings 更新设备设置
POST /api/v1/devices/{id}/commands 下发控制命令
GET /api/v1/device-commands/{id} 查询命令与时间线
GET /api/v1/device-scenes 获取自动化场景
POST /api/v1/device-scenes 创建自动化场景
POST /api/v1/device-scenes/{id}/execute 执行场景
POST /api/v1/devices/{id}/firmware-upgrades 发起固件升级

数据层新增了设备命令记录和自动化场景记录，命令状态支持：

awaiting_confirmation -> queued -> sent -> acknowledged -> succeeded

失败状态也会保留原因，便于后续重试和排障。

# 五、代码结构调整

Flutter 端新增独立的 features/devices 领域目录，按职责拆分：

  • data：接口和仓库。
  • application：设备状态和业务流程。
  • view：设备中心、详情、命令和新增设备页面。
  • panels：三类设备的专用控制面板。
  • widgets：通用状态、操作和反馈组件。

首页和“我的”页面只负责提供入口，不再继续堆叠设备控制逻辑。

后端把设备命令和场景记录纳入数据库模型，避免所有状态只存在于页面内存中。

本轮完成了代码、接口和实际页面验证：

实际测试中，未确认的收伞命令停留在 awaiting_confirmation；确认后依次经过排队、发送、设备确认和执行成功，页面能够查询同一个命令编号的完整时间线。

# 七、已完成范围与下一阶段

本轮已经完成批准方案中的软件产品闭环，包括页面、交互、状态模型、接口、数据库、模拟网关和多端验证。

当前 MQTT、BLE 和具体硬件厂商协议仍然是预留适配层。这部分必须结合真实钓箱、钓伞、钓台的通信协议、硬件安全约束和实机测试继续实施，不能用页面模拟代替硬件验收。

下一阶段建议按以下顺序推进：

  1. 选择一款真实设备完成 MQTT 或 BLE 适配。
  2. 建立设备离线、超时、重复命令和弱网重试规范。
  3. 增加 WebSocket 或 MQTT 实时事件推送。
  4. 用真实设备完成风雨、低温、负载和急停安全测试。
  5. 根据实际用户使用记录优化默认场景和告警阈值。
