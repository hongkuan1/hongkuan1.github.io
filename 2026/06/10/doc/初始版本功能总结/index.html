# 1. 版本定位

渔趣 App v1 是一个面向垂钓场景的移动端原型/初版应用，已经形成了比较完整的前端页面骨架和主业务路径。整体产品方向可以概括为：

  • 垂钓环境与智能设备看板
  • 钓点发现、社区内容和攻略浏览
  • 渔具商城、服务和二手交易入口
  • 鱼情、动态、长文、二手内容发布
  • 用户登录、个人资料、个人统计和设置管理

当前 v1 的主要价值是“高保真演示 + 部分接口联调”。其中认证、用户资料、用户统计、头像上传、退出登录已经接入 API；首页 IoT、发现、商城、发布等业务仍以本地 mock 数据和占位页为主。

# 3. 应用入口与全局流程

# 3.1 启动流程

入口文件：lib/main.dart

启动时完成以下初始化：

  1. 初始化 Flutter binding。
  2. 初始化 AuthSession，从 SharedPreferences 读取 token。
  3. 初始化 DioClient，设置接口地址、超时和请求拦截器。
  4. 开启 edge-to-edge 沉浸式系统栏。
  5. 使用 ProviderScope 包裹应用。
  6. 使用 ScreenUtilInit 做移动端尺寸适配。
  7. 启动 MaterialApp.router，接入全局路由。

# 3.2 登录态路由控制

路由文件：lib/routes/app_router.dart

当前路由根据 AuthSession.isLoggedIn 决定初始入口：

  • 已登录：进入 /home
  • 未登录：进入 /login

全局 redirect 规则：

  • 未登录访问非登录页时，自动跳转 /login
  • 已登录访问登录页时，自动跳转 /home

这说明 v1 已经具备基础登录态保护能力。

# 5. 功能模块总结

# 5.1 登录与认证

相关文件：

  • lib/features/auth/view/login_screen.dart
  • lib/features/auth/data/auth_repository.dart
  • lib/features/auth/data/auth_models.dart
  • lib/core/auth/auth_session.dart

已实现能力：

  • 登录页具备湖景背景、毛玻璃表单、验证码登录/密码登录 Tab。
  • 密码登录已调用接口：POST /api/v1/auth/login/password。
  • 登录成功后保存 token 和 refreshToken 到本地。
  • Dio 请求拦截器会自动添加 Authorization: Bearer <token>。
  • 登录成功后跳转首页，并刷新当前用户与统计 provider。
  • 支持登录失败错误提示、提交中 loading、基础空值校验。
  • 默认填入开发演示账号：13800000000 / 123456，便于联调。

占位或未完成能力：

  • 验证码登录接口未接入，点击会提示“验证码登录接口暂未提供”。
  • 注册表单已做 UI 和兴趣选择，但没有真实注册接口。
  • 一键登录、微信/手机/桌面等社交登录仅为展示。
  • 忘记密码入口仅为静态文字。

# 9. 主要风险与缺口

# 9.1 数据真实度不足

除认证和用户模块外，大量业务数据仍是本地静态数据。首页水况、IoT 设备、发现、商城、发布内容均未接后端或真实设备。

影响：

  • 适合演示，不适合真实业务闭环。
  • 数据无法跨端、跨账号保存。
  • 页面刷新不会获取真实最新内容。

# 9.2 发布链路缺少闭环

发布页面 UI 完整，但没有表单校验、媒体上传、草稿保存、接口提交和失败重试。

影响：

  • 用户会看到“发布成功”，但后端没有真实内容。
  • 草稿、预览、位置、标签等能力无法真实使用。

# 9.3 IoT 控制仅为本地状态

设备开关只是修改 Riverpod 本地 state，没有设备指令、连接状态、失败回滚、离线禁用和超时提示。

影响：

  • 可以演示智能设备概念，但不能控制真实硬件。
  • 如果后续接真实设备，需要补充更严谨的状态机。

# 9.4 二级页面为统一演示模板

18 个二级功能页面可以支撑客户路径串讲，但不是实际业务页面。

影响：

  • 业务深度不足。
  • 无法完成真实筛选、列表、详情、订单、收藏、成就等操作。

# 9.5 平台能力需要补齐

当前 Flutter 侧定义了头像选择 MethodChannel，但在当前 iOS 工程中没有看到对应实现。

影响：

  • 在 iOS 真机上点击头像选择可能触发 MissingPluginException。
  • 需要确认是否存在未包含在当前目录的 Android 或其他平台实现。

# 10. 下一阶段建议

# P0：补齐真实业务闭环

  • 发布模块接入内容发布接口。
  • 图片/视频选择与上传形成闭环。
  • 草稿保存先落本地，再逐步接后端草稿接口。
  • 发现页接入钓点、社区动态、攻略列表接口。
  • 商城页接入商品列表、商品详情、购物车接口。
  • IoT 首页接入设备列表、设备状态和指令下发接口。

# P1：强化状态与异常处理

  • 所有 API 页面统一补齐 loading、empty、error、retry。
  • IoT 指令增加执行中、成功、失败、超时和回滚状态。
  • 发布按钮增加提交中禁用和失败提示。
  • 登录过期时统一清 token 并跳转登录页。

# P1：沉淀业务模型与 repository

建议逐步抽出以下模型和仓储层：

  • FishingSpot
  • CommunityPost
  • GuideArticle
  • Product
  • CartItem
  • Order
  • IotDevice
  • ContentPost
  • FishReport
  • SecondHandItem

页面不要长期直接维护静态数组，避免后续接口接入成本变高。

# P2：补充测试与平台能力

  • 为 AuthRepository、ApiResponse、AuthSession 补单元测试。
  • 为登录跳转、退出登录、资料编辑补 widget 测试。
  • 补齐 iOS 图片选择实现和权限说明。
  • 确认 Android 工程是否应纳入当前仓库。
  • 对发布表单做小屏溢出和输入法遮挡验证。

# 11. 总结

渔趣 App v1 已经具备比较完整的移动端产品雏形：主 Tab、登录态、首页看板、发现、商城、个人中心、设置和发布链路都已经搭好，视觉风格统一，演示路径清晰。当前最成熟的部分是认证与用户资料模块；最需要继续建设的是发布、IoT、发现、商城这几条真实业务闭环。

如果作为客户演示版本，v1 已经能够表达产品定位和核心体验；如果作为可上线版本，还需要优先补齐真实数据接口、提交链路、媒体上传、设备控制和异常状态处理。
